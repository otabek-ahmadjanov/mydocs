# Утечки памяти

Утечка памяти — это ситуация, когда в `heap` присутствуют объекты со ссылками, которые больше не используются, но сборщик мусора не может удалить их из памяти, так как он удаляет только объекты без ссылок.

**`Типы утечек памяти:`**

- **через статические поля**

  В Java *статические* поля имеют срок жизни, который обычно соответствует полному жизненному циклу запущенного приложения (за исключением случаев, когда `ClassLoader` получает право на сборку мусора).

  Если коллекции или большие объекты объявлены как *статические*, то они остаются в памяти на протяжении всего времени работы приложения, тем самым блокируя
  жизненно важную память, которую можно было бы использовать в другом месте.

- **через незакрытые ресурсы**

  Всякий раз, когда мы создаем новое соединение или открываем поток, JVM выделяет память для этих ресурсов. В качестве примера можно привести соединения с базой данных, входные потоки и объекты сессий.

  Забыв закрыть эти ресурсы, можно заблокировать память, что сделает их недоступными для GC.

- **через неправильную имплементацию `equals()` и `hashCode()`**

  При определении новых классов очень распространенной ошибкой является отсутствие надлежащих переопределенных методов для `equals()` и `hashCode()`.

  `HashSet` и `HashMap` используют эти методы во многих операциях, и если они переопределены неправильно, то могут стать источником потенциальных проблем с утечкой памяти.

- **через внутренние классы, которые ссылаются на внешние**

  Когда создаётся объект нестатического внутреннего класса, компилятор Java автоматически добавляет скрытое поле, содержащее ссылку на внешний объект.

    ```java
    class Outer {
        class Inner {
            void print() {
                System.out.println("Hello");
            }
        }
    }
    ```

  После компиляции `Inner` фактически выглядит так (упрощённо):

    ```java
    class Outer$Inner {
        private final Outer this$0; // Скрытая ссылка на внешний объект
    
        Outer$Inner(Outer outer) {
            this.this$0 = outer;
        }
    
        void print() {
            System.out.println("Hello");
        }
    }
    ```

  **`Проблема**:` если объект внутреннего класса продолжает использоваться, то внешний объект **не может быть удалён сборщиком мусора**, даже если на него больше нет явных ссылок.
  **`Решение:`** если внутреннему классу не требуется доступ к нестатическим членам внешнего класса, его следует объявить как `static`.
