# Контракт метода `equals`

`equals` должен реализовывать **отношение эквивалентности**. Это пять свойств, которые нельзя нарушать.


## 1. Пять свойств контракта

### Рефлексивность
Объект равен сам себе.
```java
x.equals(x) == true
```
Нарушить сложно даже специально.

### Симметричность
Если первый равен второму — то и второй равен первому.
```java
если x.equals(y) == true, то y.equals(x) == true
```
**Ловушка:** ваш класс «знает» о другом типе, а тот о вашем — нет.
Пример: `CaseInsensitiveString`, который пытается сравниваться с обычным `String`.
Ваш класс скажет `true`, а `String` — `false`. Симметрия сломана.

### Транзитивность
Если `x == y` и `y == z`, то `x == z`.
```java
если x.equals(y) и y.equals(z), то x.equals(z)
```
**Главная ловушка:** добавление значимого поля в подкласс.
`Point` сравнивается по `(x, y)`, а `ColorPoint` добавляет цвет — корректно совместить
сравнение родителя и потомка через наследование **невозможно** без нарушения свойств.

**Пример из книги:** `ColorPoint.equals` сравнивает координаты *и* цвет, но при сравнении
с обычным `Point` игнорирует цвет (иначе сломается симметрия). Тогда:
```java
Point      p  = new Point(1, 2);
ColorPoint cp1 = new ColorPoint(1, 2, RED);
ColorPoint cp2 = new ColorPoint(1, 2, BLUE);
 
cp1.equals(p)  == true   // цвет игнорируем
p.equals(cp2)  == true   // цвет игнорируем
cp1.equals(cp2) == false // цвета RED ≠ BLUE
```
`cp1 == p` и `p == cp2`, но `cp1 ≠ cp2` → транзитивность сломана.

Вместо `instanceOf` можно использовать `getClass` чтобы проверить, что классы имеют один и тот же класс реализации, но это нарушает **Принцип подстановки Лисков**, так как такое важное свойство типа не будет работать также для его подтипов.

**Не существует способа расширения инстанцируемого класса с добавлением компонента-значения с сохранением контракта `equals`!**

### Согласованность (consistency)
При неизменных объектах повторные вызовы дают тот же результат.
```java
многократный x.equals(y) — всегда одно и то же
```
**Вывод:** не зависеть от ненадёжных ресурсов.
Антипример из JDK — `URL`, чей `equals` лезет в сеть.

### Не-`null` (non-nullity)
Любой объект не равен `null`.
```java
x.equals(null) == false
```
Явная проверка `if (o == null)` не нужна — `instanceof` сам вернёт `false` для `null`.

## 2. Проблема наследования и транзитивности

> Центральный и самый сложный пункт главы.

**Нет способа** расширить инстанцируемый класс новым значимым полем, сохранив контракт.
Любой подход ломается:

| Подход | Что ломает |
|---|---|
| Сравнение через `instanceof` | **Симметрию** (родитель и потомок дают разные результаты) |
| Сравнение через `getClass()` | Принцип подстановки **Лисков** (подкласс перестаёт быть заменой родителя) |

**Рекомендация Блоха:** вместо наследования — **композиция** (*favor composition over inheritance*).
Сделать `Point` приватным полем `ColorPoint` и добавить метод-представление `asPoint()`.

> **Исключение:** значимые поля можно безопасно добавлять в подкласс **абстрактного** класса —
> напрямую создать экземпляр родителя нельзя, конфликта сравнений не возникает.

## 3. Рецепт качественного `equals`

```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;            // 1. == для self-проверки
    if (!(o instanceof MyClass)) return false; // 2. instanceof (+ закрывает null)
    MyClass other = (MyClass) o;           // 3. безопасный cast
    return field1 == other.field1          // 4. сравнение значимых полей
            && Objects.equals(field2, other.field2);
}
```

1. **`==` для self-проверки** — сравнить с `this`. Оптимизация для дорогих сравнений.
2. **`instanceof` для проверки типа** — не тот тип → `false`. Заодно закрывает `null`.
3. **Приведение типа** — после `instanceof` безопасно.
4. **Сравнить значимые поля.**

### Чем сравнивать поля

| Тип поля | Метод |
|---|---|
| примитивы (кроме `float`/`double`) | `==` |
| объектные ссылки | `.equals()` |
| `float` | `Float.compare()` |
| `double` | `Double.compare()` (из-за `NaN`, `-0.0`) |
| массивы | `Arrays.equals()` |
| поля, способные быть `null` | `Objects.equals(a, b)` (без NPE) |

5. **Порядок ради производительности** — сначала поля, которые чаще различаются или дешевле вычисляются.

## 4. Прочие рекомендации

- **Всегда переопределяй `hashCode`** вместе с `equals` — иначе ломается `HashMap`/`HashSet`.
- **Не подменяй сигнатуру:**
  ```java
  public boolean equals(Object o)    // ✅ правильно — переопределение
  public boolean equals(MyClass o)   // ❌ ошибка — это перегрузка
  ```
- **Используй `@Override`** — компилятор поймает ошибку сигнатуры.
- **Не умничай** — не делай «слишком умных» сравнений (эквивалентность файлов через сеть и т.п.). Сравнивай поля.

### Когда `equals` переопределять НЕ нужно
- каждый экземпляр уникален по природе (`Thread`);
- классу не нужно «логическое равенство»;
- суперкласс уже даёт подходящий `equals`;
- класс приватный/package-private и метод не вызовется.

---

## TL;DR

- equals = отношение эквивалентности: **рефлексивность · симметрия · транзитивность · согласованность · не-null**.
- Главная ловушка — **наследование + значимое поле** ломает симметрию/транзитивность → решай **композицией**.
- Рецепт: `==` → `instanceof` → `cast` → сравнение значимых полей (правильными методами).
- Всегда переопределяй **`hashCode`** вместе с equals.
- Сигнатура строго `equals(Object o)` + `@Override`.
- Не переопределяй без необходимости.
