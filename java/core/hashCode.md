# `hashCode()`


## TL;DR

- Дефолтный `hashCode()` из `Object` возвращает **identity hash code** — 31-битное псевдослучайное `int`-значение, привязанное к *экземпляру*, а не к его содержимому.
- Значение **лениво вычисляется один раз** и **кэшируется** в заголовке объекта (mark word). Дальше только читается.
- Это **не адрес объекта**: хэш обязан пережить перемещения объекта сборщиком мусора, а адрес — нет.
- Дефолтный хэш бесполезен для value-семантики: два логически равных объекта дадут разные хэши.

---

## 1. Контракт `hashCode()`

Контракт состоит из трёх пунктов:

1. **Консистентность.** В рамках одного запуска приложения повторные вызовы `hashCode()` на одном объекте должны возвращать одно и то же значение, если данные, используемые в `equals()`, не менялись. Между разными запусками значение может отличаться.

2. **Согласованность с `equals()`.** Если `a.equals(b) == true`, то `a.hashCode() == b.hashCode()` **обязательно**. Это ключевой пункт — именно его нарушает дефолтная реализация при переопределённом `equals()`.

3. **Необязательность (но желательность) различия.** Если `a.equals(b) == false`, хэши *не обязаны* различаться. Но ради производительности хэш-таблиц желательно, чтобы неравные объекты давали разные хэши (хорошее распределение).

### Почему нарушение критично

`HashMap`, `HashSet`, `Hashtable` сначала ищут по хэшу (bucket), и только внутри бакета сравнивают через `equals()`. Если равные объекты имеют разные хэши, они попадут в разные бакеты — и коллекция «не найдёт» уже добавленный элемент.

```java
Map<PhoneNumber, String> m = new HashMap<>();
m.put(new PhoneNumber(707, 867, 5309), "Jenny");
m.get(new PhoneNumber(707, 867, 5309)); // → null, если hashCode() не переопределён!
```

## 2. Дефолтная реализация в `Object`

Концептуально выглядит так:

```java
public int hashCode() {
    // identity hash code — зависит от экземпляра, а не от содержимого
}
```

Свойства дефолтного значения:

- Зависит от конкретного объекта, а не от полей → два разных объекта с одинаковыми полями дают **разные** хэши.
- Постоянно в течение жизни объекта (в одном запуске JVM), но не гарантируется между запусками.
- Получить «сырое» значение независимо от переопределений: `System.identityHashCode(obj)`.

---

## 3. Как это устроено под капотом (HotSpot)

### Где хранится

У каждого объекта есть заголовок (object header), а в нём — **mark word** (машинное слово, 64 бита на 64-битной JVM). Identity hash code кэшируется именно здесь.

- Под хэш выделен **31 бит** (не 32) → значение неотрицательное, до 2³¹−1.
- Хэш **лениво вычисляется и кэшируется** в mark word.

### Почему это НЕ адрес объекта

HotSpot использует **перемещающий (compacting) GC** — объекты физически переезжают в памяти. Контракт требует стабильности хэша в течение жизни объекта. Если бы хэш был живым адресом:

```
объект по адресу 0x1000 → hashCode() = 0x1000
   ... GC перемещает объект ...
объект теперь по адресу 0x8000 → hashCode() = 0x8000   ← контракт нарушен!
```

Поэтому значение вычисляется один раз и **фиксируется** в заголовке. При переезде mark word (а с ним и хэш) едет вместе с объектом.

---

## 4. Процесс по шагам

1. Объект создаётся. Поле хэша в mark word **пустое** (ленивая инициализация).
2. Первый вызов `hashCode()` / `System.identityHashCode(obj)`.
3. JVM видит, что хэш не установлен → вызывает `get_next_hash()`.
4. `get_next_hash()` генерирует значение по алгоритму (дефолт `=5`: адрес не участвует, чистый псевдослучайный генератор).
5. Значение записывается в mark word → закэшировано.
6. Последующие вызовы просто **читают** значение из заголовка. Генерация больше не повторяется.
7. При GC объект переезжает — mark word едет вместе с ним, хэш неизменен.

---

## 7. Практические выводы

- Дефолтный `hashCode()` привязан к **идентичности** экземпляра, а не к **содержимому** → для value-объектов бесполезен.
- **Переопределяя `equals()`, всегда переопределяй `hashCode()`** — иначе объект ломается в хэш-коллекциях.
- Хорошая реализация `hashCode()` для value-класса использует те же поля, что и `equals()`, комбинируя их (например, `Objects.hash(...)` или ручная схема `31 * result + c` для горячих путей).
- `System.identityHashCode(obj)` даёт исходный identity hash независимо от переопределений — полезно для отладки (проверить, один ли это физический объект).

### Пример корректной реализации

```java
@Override
public int hashCode() {
    int result = Short.hashCode(areaCode);
    result = 31 * result + Short.hashCode(prefix);
    result = 31 * result + Short.hashCode(lineNum);
    return result;
}

// или короче, ценой небольшой производительности:
@Override
public int hashCode() {
    return Objects.hash(areaCode, prefix, lineNum);
}
```

> Ключевая интуиция всей темы: **хэш обязан пережить переезды объекта, а адрес — нет. Значит хэш не может быть адресом; в лучшем случае он мог когда-то от него произойти.**
